# Configuration

## Config Files

| File | Location | Purpose |
|------|----------|---------|
| `blockhost.yaml` | `/etc/blockhost/` | Server keypair, public secret, admin wallet, fund manager settings |
| `web3-defaults.yaml` | `/etc/blockhost/` | Blockchain config (Blockfrost, network, policy IDs, validator address) |
| `admin-commands.json` | `/etc/blockhost/` | Admin command definitions (port knocking, etc.) |
| `addressbook.json` | `/etc/blockhost/` | Role-to-wallet mapping (admin, server, hot, dev, broker) |
| `revenue-share.json` | `/etc/blockhost/` | Revenue sharing configuration (dev/broker splits) |
| `vms.json` | `/var/lib/blockhost/` | VM database (IPs, NFT state, ownership) |
| `engine.json` | `/usr/share/blockhost/` | Engine manifest (identity, wizard plugin, constraints) |

## web3-defaults.yaml

Blockchain configuration for the Cardano engine:

```yaml
blockchain:
  blockfrost_project_id: preprodXXXXXXXXXXXXXXXXXXXXXXXXXX
  network: preprod                    # mainnet | preprod | preview
  nft_policy_id: <56-char hex>        # NFT minting policy ID
  subscription_validator_hash: <56-char hex>   # Script hash of subscription validator
  beacon_policy_id: <56-char hex>     # Beacon minting policy ID
  subscription_validator_address: addr_test1...  # Bech32 validator address
```

All `*_policy_id` and `*_validator_hash` values are 56-character hex strings (28 bytes). The `subscription_validator_address` is the bech32 enterprise address derived from the subscription validator script hash.

Networks:
- `mainnet` — Cardano mainnet (`cardano-mainnet.blockfrost.io`)
- `preprod` — Pre-production testnet (`cardano-preprod.blockfrost.io`)
- `preview` — Preview testnet (`cardano-preview.blockfrost.io`)

## blockhost.yaml

General server and fund manager configuration:

```yaml
server_public_key: <hex secp256k1 public key>   # For ECIES encryption of user data
public_secret: blockhost-access                  # Text used as signing challenge prefix

admin:
  credential_nft_id: 0          # Token ID of the admin access credential NFT
  wallet_address: addr1...      # Admin wallet address (for command detection)
  shared_key: <32-byte hex>     # HMAC key for metadata admin commands
  max_command_age: 86400        # Seconds; nonces older than this are pruned

fund_manager:
  fund_cycle_interval_hours: 24
  min_withdrawal_lovelace: 50000000        # 50 ADA minimum before collection
  gas_low_threshold_lovelace: 5000000     # 5 ADA — triggers warning
  hot_wallet_gas_lovelace: 5000000        # 5 ADA target hot wallet balance
  server_stablecoin_buffer_lovelace: 5000000  # Token base units
```

## Addressbook

Role-to-wallet mapping in `/etc/blockhost/addressbook.json`. All addresses are bech32:

```json
{
  "admin":  { "address": "addr1..." },
  "server": { "address": "addr1...", "keyfile": "/etc/blockhost/deployer.key" },
  "dev":    { "address": "addr1..." },
  "broker": { "address": "addr1..." }
}
```

Entries with `keyfile` can sign transactions. The `hot` entry is auto-added on first fund cycle.

The `keyfile` for the `server` entry stores a BIP39 mnemonic (24 words), not a hex private key. The wallet derives CIP-1852 keys from the mnemonic at runtime.

## deployer.key

The server wallet's key material. Contains a 24-word BIP39 mnemonic (space-separated). Example:

```
abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon abandon art
```

Generated by `keygen.ts` via the root agent's `generate-wallet` action. Stored at `/etc/blockhost/deployer.key` (chmod 600, readable by `blockhost` group).

## Revenue Sharing

Configured in `/etc/blockhost/revenue-share.json`:

```json
{
  "enabled": true,
  "total_bps": 100,
  "recipients": [
    { "role": "dev",    "bps": 50 },
    { "role": "broker", "bps": 50 }
  ]
}
```

`total_bps` is basis points (100 bps = 1%). `recipients[].role` maps to addressbook keys. The recipient `bps` values must sum to `total_bps`. The legacy `total_percent` / `percent` format is also accepted (converted to bps internally).

## admin-commands.json

Defines the commands available for on-chain admin dispatch (transaction metadata protocol):

```json
{
  "commands": {
    "knock-80": {
      "action": "knock",
      "params": {
        "port": 80,
        "proto": "tcp",
        "duration_seconds": 300
      }
    }
  }
}
```

`action` must match a registered action handler in `src/admin/index.ts`. Currently supported: `knock` (opens an iptables rule for a port for a configured duration).
